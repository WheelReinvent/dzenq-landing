01 — What is DzenQ

A "Thank You"
that cannot be denied

DzenQ turns every expression of gratitude into a tamper-proof digital certificate. It can be any digital media — a document, image, audio, or video — carrying the recipient's identity and a cryptographic signature that locks everything together.

No server, no company, no authority can alter or erase it. The certificate lives on your device — shareable yet immutable, like an inscription in stone.

  • 02.Contains a SHA-256 hash of every byte of content
  • 03.Verifiable offline — no internet required to confirm authenticity
  • 04.Designed to outlast any platform, company, or cloud service

05 — Technology

The architecture
of permanence

Decentralised Identity

Each user controls a keypair generated locally. Your public key is your identity — no username, no email, no account. Optionally anchor to a DID for interoperability.

No personal data is stored anywhere. The identity is yours, portable, and sovereign.

Our approach

Local keypair generation with optional hardware key backup and Shamir's Secret Sharing recovery.

DzenQ Identity Root
├─ Public Key // Ed25519, 32 bytes
├─ Private Key // never leaves device
├─ DID Document // optional W3C DID
│ └─ Verification Method // JWK2020
└─ Recovery Scheme // Shamir SSS

Where Certificates Live

Certificates are self-contained JSON-LD files. They live on your device first. Optional anchoring to IPFS or a personal server gives redundancy without dependency.

Local-first principle

No cloud required. Works offline forever. You control your data completely.

certificate-0042.dzenq
├─ @context // JSON-LD schema
├─ issuer // sender DID
├─ recipient // receiver DID
├─ payload_hash // SHA-256
└─ signature // Ed25519

Ed25519 Signing

Ed25519 provides 128-bit security with 64-byte signatures and sub-millisecond signing. Standardised in RFC 8032, supported by every major cryptographic library and hardware key.

Quantum migration plan

Certificate format reserves a field for ML-DSA / Dilithium. Migration tooling will allow re-signing existing certificates when ready.

Signing Process
1. Serialise payload // canonical JSON
2. SHA-256 hash // 32 bytes
3. Ed25519 sign // private key op
4. Embed signature // base64url
5. Export .dzenq // self-contained

Verification Without Trust

Anyone with the sender's public key can verify a certificate. No internet. No DzenQ servers. No trust in any third party. Pure mathematics.

What verification guarantees

Content unchanged since signing · Signed by claimed identity · Timestamp locked on signing key

Verification Steps
1. Load .dzenq file // any device
2. Hash payload // reproduce SHA-256
3. Fetch public key // from DID or local
4. Ed25519 verify // mathematical proof
5. ✔ Valid — or — ✗ Tampered
